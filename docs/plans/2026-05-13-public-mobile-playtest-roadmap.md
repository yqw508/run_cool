# 萌娃酷跑移动端试玩版本计划

## 目标

当前阶段的核心目标不是先做微信小程序审核上线，而是先把《萌娃酷跑》部署成一个朋友能在手机上点开就玩的 H5 游戏。

理想体验是：朋友收到一个链接或二维码，在微信/浏览器里打开，10 秒内进入游戏，能选角色、选关卡、玩一局，并愿意转发给其他朋友试玩。

## 当前项目状态

已具备：

- Phaser + Vite H5 游戏工程。
- 手机竖屏 390 x 720 游戏画布。
- 角色选择。
- 关卡选择：校园、商场、动物园、游乐园。
- 三车道跑酷基础玩法。
- 星星收集、障碍物、跳跃、滑行。
- 生命值机制：撞障碍扣血，不立即结束。
- 受击闪烁、震动、短暂无敌。
- 每个场景有独立轻量背景音乐入口。
- 单元测试覆盖输入、进度、关卡、生命值逻辑。

主要不足：

- 还没有公网部署地址。
- 还没有正式分享页、二维码和分享文案。
- 美术与音频仍偏原型，缺少成品感。
- 没有新手引导、暂停、明确通关目标。
- 没有试玩反馈收集机制。

## 版本路线

### V0.1 当前原型版

**目标**：本地可玩，验证核心跑酷循环。

已完成内容：

- 角色选择。
- 关卡选择。
- 基础跑酷。
- 生命值和受击反馈。
- 四个主题场景。

验收标准：

- `npm.cmd run test` 通过。
- `npm.cmd run build` 通过。
- 本地浏览器可打开并完成一局游戏。

### V0.2 GitHub 公网试玩版

**目标**：通过 GitHub 部署一个可分享链接，让朋友在手机端试玩。

推荐部署方式：

- 代码托管到 GitHub。
- 使用 GitHub Pages、Vercel 或 Netlify 部署 H5。
- 优先建议 Vercel 或 Netlify，因为 Vite 项目部署更顺，预览链接也方便。

需要完成：

| 编号 | 任务 | 复杂度 | 说明 |
| --- | --- | --- | --- |
| T-01 | 初始化 Git 仓库并提交当前代码 | S | 包含源码、docs、package-lock，不提交 `node_modules` |
| T-02 | 创建 GitHub 仓库并推送代码 | S | 仓库名建议 `mengwa-runner` 或 `run-cool` |
| T-03 | 配置公网部署 | S | 推荐 Vercel/Netlify；GitHub Pages 也可 |
| T-04 | 增加基础分享说明 | S | README 写清试玩链接、手机打开方式、反馈方式 |
| T-05 | 手机端冒烟测试 | S | 微信内置浏览器、iOS Safari、Android 浏览器至少各测一次 |

验收标准：

- 朋友能通过 HTTPS 链接打开游戏。
- 首屏能看到“萌娃酷跑”和角色/关卡选择。
- 一局游戏能开始、受击、结算、重新开始。
- 链接在微信聊天里可正常打开。

### V0.3 朋友试玩优化版

**目标**：让非开发者朋友和孩子能更容易上手，并收集反馈。

需要完成：

| 编号 | 任务 | 复杂度 | 说明 |
| --- | --- | --- | --- |
| T-06 | 增加新手引导 | M | 第一次进入显示左右滑、上滑、下滑提示 |
| T-07 | 增加暂停/继续 | S | 移动端误触或家长中断时需要 |
| T-08 | 明确每局目标 | M | 例如每关 300 米，跑完显示通关 |
| T-09 | 优化结算页 | M | 区分通关/失败，显示星级、距离、鼓励文案 |
| T-10 | 增加反馈入口 | S | README 或页面上放反馈二维码/问卷链接 |
| T-11 | 首屏加载优化 | S | 加载提示、资源压缩、避免白屏 |

验收标准：

- 没看说明的人能在 30 秒内开始第一局。
- 孩子失败后愿意再玩一局。
- 试玩者能清楚说出“最喜欢/最不懂/最难”的点。

### V0.4 视觉与音频体验版

**目标**：从功能原型提升到“像一个儿童游戏”的体验。

需要完成：

| 编号 | 任务 | 复杂度 | 说明 |
| --- | --- | --- | --- |
| T-12 | 角色正式素材 | L | 至少 2-3 个主角有统一风格立绘/动画 |
| T-13 | 场景正式背景 | L | 校园、商场、动物园、游乐园分层背景 |
| T-14 | 障碍物正式图标 | M | 书包、栏杆、购物袋、爆米花、栅栏等 |
| T-15 | UI 视觉统一 | M | 按钮、生命值、星星、结算页统一风格 |
| T-16 | 正式 BGM 和音效 | M | 每个场景独立音乐，收集/跳跃/受击/通关音效 |

验收标准：

- 首屏能明显看出这是《萌娃酷跑》，不是通用跑酷 demo。
- 四个关卡不靠文字也能大致识别。
- 音乐和音效不会刺耳，适合孩子反复玩。

### V0.5 渠道扩展版

**目标**：根据试玩反馈决定是否进入微信小游戏/小程序或其他分发渠道。

候选渠道：

- H5 长期部署：最轻量，适合朋友传播。
- PWA：让手机网页更像 App，可添加到桌面。
- 微信小游戏：适合长期运营、排行榜、群分享、收藏入口。
- 微信小程序 WebView：迁移快，但游戏体验和限制不如小游戏自然。

建议决策标准：

- 如果只是朋友间试玩和家庭传播：继续 H5。
- 如果有持续更新、排行榜、活动和更强微信传播需求：考虑微信小游戏。
- 如果要做品牌展示页 + 游戏入口：H5/PWA 更灵活。

## 测试策略

自动化测试：

- `npm.cmd run test`
- `npm.cmd run build`

手动测试：

- 微信内置浏览器打开试玩链接。
- iPhone Safari 打开试玩链接。
- Android 浏览器打开试玩链接。
- 进入首页后选择不同角色和关卡。
- 至少完成一次受击、一次结算、一次重新开始。
- 打开声音，确认背景音乐和受击音效不刺耳。

## 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 微信内置浏览器音频自动播放限制 | 音乐可能需要用户点击后才能播放 | 将音频启动放在“开始酷跑”点击之后 |
| Phaser 包体较大 | 首屏加载慢 | 后续做加载页、资源压缩、CDN 部署 |
| 低端安卓性能不足 | 卡顿影响孩子体验 | 控制粒子、动画和图片尺寸，真机测试 |
| 美术原型感较强 | 朋友觉得不够成品 | V0.4 集中补视觉与音频 |
| GitHub Pages 路径问题 | 部署后资源 404 | Vite 配置 `base` 或改用 Vercel/Netlify |

## 回滚策略

- 每个版本单独打 Git tag，例如 `v0.2-playtest`。
- 部署平台保留历史部署记录。
- 如果新版出现明显问题，回滚到上一个可玩的部署版本。
- 如果玩法改动争议大，保留旧版本链接给试玩者对比。

1. 初始化 Git 仓库。
2. 添加 `.gitignore`。
3. 提交当前代码和文档。
