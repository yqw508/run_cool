# 3D 角色跑酷实现方案

## 1. 背景与目标

当前项目已经完成了 H5 跑酷主流程：角色大厅、地图选择、三跑道跑酷、障碍物、收集物、移动端滑动操作。但角色表现仍以二维图片为主，即使用裁剪分层和呼吸动画，也很难达到《汤姆猫跑酷》《地铁跑酷》那类产品的角色质感。

新的目标是把视觉核心升级为 **Q 版 3D 角色 + 骨骼动画 + 2.5D/3D 场景表现**，同时保留当前项目“打开链接就能玩、方便分享给朋友”的优势。

### 体验目标

- 角色大厅像 RPG 角色展示场景：角色站在花园或主题场景中，有待机呼吸、转身、专属动作。
- 跑酷视角使用正向三跑道背后视角，角色朝前跑，动作包括跑步、跳跃、下滑、左右变道、受击、胜利、失败。
- 障碍物和道路有明显空间感，玩家能从视觉上判断跳跃、下滑、换道。
- 移动端优先，微信内置浏览器、iOS Safari、Android Chrome 都能试玩。
- 仍然可以通过一个 H5 链接或二维码分享给朋友。

## 2. 技术路线结论

推荐路线：**Vite H5 + Three.js 3D 表现层 + GLB 角色资产 + 当前游戏逻辑逐步迁移**。

不建议继续投入二维图片角色优化。二维分层适合轻量养成或卡牌展示，但跑酷场景需要背后视角、身体朝向、脚底接触、跳跃和下滑动作，最终仍会受限。

### 技术选型

| 模块 | 推荐方案 | 说明 |
| --- | --- | --- |
| 前端工程 | 继续使用 Vite | 当前项目已是 Vite，静态部署和分享成本低 |
| 3D 渲染 | Three.js | H5 生态成熟，适合加载 GLB、移动端 WebGL |
| 角色模型 | GLB/glTF | 支持骨骼、材质、动画，一份模型可包含多个动作 |
| 动画系统 | Three.js AnimationMixer | 播放 idle/run/jump/slide 等动作 |
| 物理/碰撞 | 先保留轻量逻辑碰撞，后续再引入 Rapier 或自研胶囊体检测 | 跑酷碰撞不需要一开始就上完整物理引擎 |
| 场景 | 低面数 3D 场景块 + 远景贴图 | 保证移动端性能 |
| 发布 | 静态站点部署 | `npm.cmd run build` 生成 `dist` 后发布 |

## 3. 总体架构

### 当前架构

```text
Vite
└─ Phaser RunnerScene
   ├─ 2D 大厅
   ├─ 2D 地图选择
   ├─ 2.5D 跑道
   ├─ PNG 角色
   └─ Phaser 碰撞和输入
```

### 目标架构

```text
Vite
├─ GameApp
│  ├─ GameState：角色选择 / 地图选择 / 跑酷 / 结算
│  ├─ InputController：触摸、鼠标、键盘
│  ├─ RunController：三跑道、速度、分数、生成规则
│  ├─ CollisionController：逻辑碰撞、收集、受击
│  └─ UIOverlay：按钮、分数、暂停、结算
│
└─ ThreeRenderer
   ├─ Scene / Camera / Lights
   ├─ CharacterRig：GLB 角色、骨骼动画、动作切换
   ├─ TrackRenderer：三跑道、地面、远景
   ├─ ObstacleRenderer：栏杆、拱门、路障、滑行障碍
   └─ FXRenderer：星星、受击、选中光圈、尘土
```

第一阶段可以保留 Phaser 作为已有主流程容器，但建议尽快把渲染和控制逻辑拆开，避免后续 Phaser 2D 和 Three.js 3D 深度耦合。

## 4. 角色资产规范

每个角色最终交付一个 GLB 文件，建议目录如下：

```text
src/assets/characters3d/
  baby-brother/
    baby-brother.glb
    preview.png
    config.json
  little-star/
    little-star.glb
    preview.png
    config.json
```

### GLB 必备内容

每个角色模型需要包含：

- 低面数 Q 版模型，移动端优先。
- 骨骼绑定。
- 统一朝向：默认面向 Z 轴负方向或项目约定方向。
- 统一比例：站立角色身高约 1.0 单位，宝宝爬行角色按同等占屏面积处理。
- 动作片段：
  - `idle`：待机呼吸
  - `run`：跑步
  - `jump`：跳跃
  - `slide`：下滑/翻滚/钻过
  - `lane_left`：向左变道
  - `lane_right`：向右变道
  - `hit`：受击
  - `win`：胜利
  - `lose`：失败
  - `select`：大厅选中展示动作

### 角色配置示例

```json
{
  "id": "little-star",
  "name": "小星星",
  "model": "/assets/characters3d/little-star/little-star.glb",
  "scale": 1.0,
  "lobbyPosition": [-1.2, 0, 0.4],
  "runYOffset": 0,
  "animations": {
    "idle": "idle",
    "run": "run",
    "jump": "jump",
    "slide": "slide",
    "select": "swing"
  }
}
```

## 5. 场景与玩法表现

### 角色大厅

大厅不再摆 PNG，而是加载多个 GLB 角色模型：

- 场景是一个花园或乐园小广场。
- 每个角色站在固定区域，不重叠。
- 未选中角色播放自己的 idle 动作。
- 选中角色时：
  - 镜头轻微推近。
  - 角色播放 `select` 动作。
  - 角色脚下出现光圈或粒子。
  - 详情 UI 出现在底部，不遮挡角色主体。

### 跑酷场景

跑酷场景使用三跑道 3D 透视：

- Camera 固定在角色后上方。
- 玩家控制的是 lane index，而不是自由移动。
- 角色在 X 轴三条跑道间切换。
- 世界向角色方向滚动，或者角色向前跑、场景循环生成。
- 障碍物按距离生成，接近时用图标或形态表达操作：
  - 低障碍：栏杆、矮箱、地面玩具，要求跳跃。
  - 高障碍：横杆、拱门、飘带，要求下滑。
  - 占道障碍：树丛、路锥、售货车，要求换道。

### UI 原则

- 跑步中尽量不用文字。
- 分数、星星、生命、暂停使用图标。
- 新手引导只在首次进入时显示，后续隐藏。
- 避免顶部大面积遮挡跑道。

## 6. 实施阶段

### 阶段 1：3D 技术验证

目标：在当前 H5 项目里跑通一个 GLB 角色。

任务：

| 编号 | 任务 | 复杂度 | 依赖 |
| --- | --- | --- | --- |
| T-01 | 引入 Three.js、GLTFLoader、基础渲染循环 | M | - |
| T-02 | 新建 `ThreeRenderer`，能在页面中渲染一个 3D 场景 | M | T-01 |
| T-03 | 加载一个测试 GLB 角色并播放 idle/run 动画 | M | T-02 |
| T-04 | 在移动端视口验证帧率和触摸兼容 | S | T-03 |
| T-05 | 保留当前 Phaser 版本作为 fallback 分支 | S | - |

验收：

- 手机浏览器能看到 3D 角色。
- idle/run 动画能切换。
- 30 FPS 以上可接受，目标 45-60 FPS。
- `npm.cmd run build` 通过。
- `npm.cmd run test` 通过。

## 7. 部署与分享方案

### 方案 A：Vercel / Netlify

适合快速给朋友试玩。

流程：

1. GitHub 仓库保持最新。
2. 平台连接 GitHub 仓库。
3. Build Command 配置为：

```bash
npm run build
```

4. Output Directory 配置为：

```bash
dist
```

5. 部署完成后获得 HTTPS 链接。
6. 用二维码工具把链接生成二维码，发给朋友扫码试玩。

优点：

- 部署快。
- 每次 push 可以自动发布。
- HTTPS 默认支持，手机访问方便。

风险：

- 国内网络访问可能不稳定。
- 免费额度和平台策略可能变化，需要定期确认。

### 方案 B：GitHub Pages

适合公开展示和低成本分享。

注意点：

- Vite 项目如果部署在子路径，需要配置 `base`。
- GitHub Pages 对静态资源路径较敏感，必须用实际线上路径验证。
- 大型 3D 资产不建议全部首屏加载，应按需加载。

### 方案 C：腾讯云 COS + CDN

适合面向国内朋友稳定分享。

流程：

1. 本地或 CI 执行 `npm.cmd run build`。
2. 上传 `dist` 到 COS Bucket。
3. 配置 CDN 加速和 HTTPS。
4. 使用自定义域名，例如：

```text
https://play.example.com
```

优点：

- 国内访问更稳。
- 适合后续小范围灰度试玩。

风险：

- 需要云资源配置。
- CDN 缓存策略要处理好，否则朋友可能看到旧版本。

## 8. 维护策略

### 版本管理

建议版本号格式：

```text
v0.1-2d-prototype
v0.2-3d-lobby
v0.3-3d-runner
v0.4-character-polish
v0.5-playtest
```

每次可分享版本都打 Git tag，方便回滚。

### 资产管理

3D 资产要和代码一样管理规范：

- 原始工程文件放 `docs/assets/source` 或单独资产仓库。
- 游戏运行只引用压缩后的 `.glb`。
- 每个角色有 `config.json`，避免把模型缩放、动画名写死在代码里。
- 大文件不直接频繁改名，避免 CDN 缓存混乱。

### 性能预算

移动端建议：

- 首屏加载控制在 10 MB 以内，越低越好。
- 单个角色 GLB 控制在 1-5 MB。
- 贴图优先 512 或 1024，避免大量 2K/4K。
- 同屏角色大厅可以分批加载，优先加载选中角色。
- 跑酷中只保留必要灯光，避免实时阴影过重。

### 监控与反馈

试玩阶段至少记录：

- 手机型号。
- 浏览器类型。
- 首次加载时间。
- 是否黑屏、白屏、卡顿。
- 是否能正常开始跑酷。
- 跳跃、下滑、换道是否灵敏。

可以先用简单反馈表，不必一开始接复杂埋点。

## 9. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| GLB 模型过大 | 首屏加载慢，微信内打不开 | Draco/Meshopt 压缩、贴图压缩、分批加载 |
| 动作不统一 | 不同角色跑步节奏怪 | 统一骨骼命名和动作命名 |
| 移动端 WebGL 性能不足 | 卡顿、发热 | 控制面数、减少灯光、关闭高成本后处理 |
| 当前 Phaser 结构耦合 | 接入 Three.js 后难维护 | 抽离 GameState、InputController、RunController |
| 部署后缓存旧资源 | 朋友看到旧版本 | 文件 hash、CDN 缓存规则、版本号检查 |
| 美术生产周期长 | 迟迟无法替换正式角色 | 先用一套标准 Q 版模型验证动作，再逐个换角色 |

## 10. 回滚策略

- 保留当前 2D/2.5D 版本作为稳定分支或 tag。
- 3D 功能通过配置开关启用：

```ts
const ENABLE_3D_RENDERER = true;
```

- 如果 3D 首屏加载失败，回退到轻量 2D 角色或显示“低配模式”。
- 每次发布前保留上一版线上 `dist`，出现问题可直接回滚静态文件。

## 11. 推荐下一步

建议不要一次性制作 5 个正式 3D 角色。先做一个技术样板：

1. 找一个临时 Q 版 GLB 角色，跑通 Three.js 加载和动画。
2. 把大厅中的一个角色替换成 3D。
3. 把跑酷中的玩家角色替换成 3D 背后视角。
4. 手机端确认帧率、加载、操作手感。
5. 再开始正式生产 5 个角色模型。

第一批里最适合做样板的是 **小星星**：

- 站立角色，比宝宝爬行更容易验证跑酷动作。
- 背带裤造型明确。
- 大厅可以做荡秋千或挥手动作。
- 跑酷中可以复用标准跑步、跳跃、下滑动作。
